Architecture

From features to response — one forward pass.

Transformer WGM 2 layers · dim 64 · 2 heads · FFN 256 Dense → LSTM×3 → Dense Weights generated, not stored

Click once per stage; deliver one sentence each. Stage 0 (shown on entry): Seven building features enter the model — height, width, depth, damping ratio, density, floor count, and fundamental frequency (derived analytically by the Rayleigh method for the 9k dataset). Stage 1 (click 1): Instead of treating the feature vector as one dense input, each scalar is individually embedded as a token with a learned positional encoding, so the model sees every feature as its own representation. Stage 2 (click 2): A 2-layer Transformer encoder — embedding dimension 64, two attention heads — applies self-attention across all feature tokens and models cross-feature interactions explicitly before anything is generated. Stage 3 (click 3): Tokens are mean-pooled and linearly projected to produce the full set of layer-weight tensors; the building description alone generates every parameter of the response network. Stage 4 (click 4): Generated weights load a conditional Dense → LSTM × 3 → Dense response network — no stored weights; the building features govern every gate, every affine map inside the network. Stage 5 (click 5): The network unrolls over the 60-second wind record and produces floor displacements at 20 normalized heights in a single forward pass — one trained model, any building, no retraining.

6.83 M parameters · population mean NRMSE: 1.05 % (10-story) · 1.01 % (15-story) · 0.88 % (20-story) — error is uniform across story counts

The 9,000-building expanded dataset has buildings with 10, 15, or 20 stories (three floor-count levels in the full-factorial grid). The naive approach — zero-padding all buildings to a 20-floor vector — forces the model to learn that the top 10 outputs are identically zero for a 10-storey building. That wastes half the output capacity and makes no physical sense. The adopted solution: linearly interpolate each building's N-storey response onto a common 20-point normalized-height grid (0 = ground level, 1 = rooftop). Every one of the 20 output points receives a physically valid response value regardless of floor count. The evaluation code reports NRMSE over all 20 normalized outputs with no masking required. Result: the interpolation model has 6,828,964 parameters and achieves a mean NRMSE of 0.98% across 900 test buildings — better than the zero-padded variant at essentially the same architecture cost.

Mean NRMSE 0.98%, median 0.89% — comfortably under 1% on a diverse portfolio. 2. HEIGHT UNIFORMITY: The height-normalized interpolation scheme maps any 10/15/20 storey building onto 20 canonical height levels. Error is nearly identical across the three height classes (1.05/1.01/0.88%) — the interpolation removes the floor-count dependency. 3. DURATION GENERALIZATION (separate test — different from the 9k NRMSE above): The network was trained only on 60-second wind records. When tested on 600-second signals (10× longer, never seen in training), RMSE = 0.040 — only a moderate increase despite a tenfold extension in duration. The model captures structural dynamics rather than memorising a fixed temporal template. (RMSE 0.040 is a raw displacement error in the same units as the response; it is NOT the same metric as the NRMSE % headline — intentionally kept off the slide to avoid confusion.) HEADLINE: single-core OpenSees takes ~80 s per building. The HyperNetwork with batched GPU inference is ~7,000× faster. PORTFOLIO IMPLICATION: 900 buildings × 80 s = 72,000 s ≈ 20 hours in OpenSees (single core). HyperNetwork completes the same portfolio in seconds — enabling real-time digital-twin updates and disaster preparedness screening. The EMI.speedup animation shows a proportional two-track comparison — not 7,000 literal bars, but a visual that conveys the scale of the gap. Note for Q&A: speedup is against single-core OpenSees. Multi-core parallelism would narrow the gap, but batched GPU inference over a full portfolio still wins decisively on total throughput.

Looking ahead

Future directions.

  • More complex loads — the immediate next step. Extend to blast, multi-hazard, and arbitrary broadband excitations.
  • Physics-informed / knowledge-enhanced training — embed governing equations to improve extrapolation and cut data demand for larger, more diverse inventories.
  • Richer features + 3D models — image and BIM-driven building representations; extend to 3D FEM with torsional and complex nonlinear behavior.
Keep this slide forward-looking and light. The model is a strong working foundation; these are natural, well-motivated next steps. Most immediate: extending to more complex loads — blast, multi-hazard, and arbitrary broadband excitations. The current training distribution covers wind, constant, and sine; broadening the load variety is the natural near-term step. Physics-informed / knowledge-enhanced training (KEDL-style, Wang 2020) would embed governing equations into the training objective, improving extrapolation and reducing data demand. This is future work — the current model is a pure time-domain Huber-loss objective. Richer features (images, BIM) and 3D structural models are longer-horizon directions that would substantially broaden the applicability of the framework.
